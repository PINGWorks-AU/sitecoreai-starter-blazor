## App.razor
In Blazor apps the `App.razor` file is typically used as the entry point, with the wrapping HTML elements. The code below shows a template `App.razor` for use with the Blazor SDK. The version shipped in the Starter Kit also pulls in a few site-specific resources; what is listed here is the minimum the SDK needs.

```html
<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="utf-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1.0" />
    <base href="/" />
    <ResourcePreloader />

    <!-- use 'Assets' array to ensure thumbprinting; see https://learn.microsoft.com/en-us/aspnet/core/blazor/fundamentals/static-files -->
    <link rel="stylesheet" href="@Assets["app.css"]" />
    <link rel="stylesheet" href="@Assets["SitecoreAI_Blazor_Starter.styles.css"]" />
    <ImportMap />
    <HeadOutlet />

    <!-- outlets used by the SDK -->
    <SectionOutlet SectionName="HeadLayout" />
    <SectionOutlet SectionName="HeadPage" />
    <SectionOutlet SectionName="SitecoreEditingScripts" />
</head>
<body>
    <Routes />
    <ReconnectModal />
    <script src="@Assets["_framework/blazor.web.js"]"></script>

    <!-- outlets used by the SDK -->
    <SectionOutlet SectionName="ScriptLayout" />
    <SectionOutlet SectionName="ScriptPage" />
</body>
</html>
```

The second stylesheet (`SitecoreAI_Blazor_Starter.styles.css` in the snippet above) is the component-scoped CSS bundle generated by Blazor. Its filename matches your assembly name with spaces replaced by underscores, so update it if you rename the project.

A `<SectionOutlet />` element can only be assigned once from within the hierarchy of components making up the Blazor page, that is, if two components write to a SectionOutlet with the same SectionName, only the content from the last one will come through.

For this reason we add the following SectionOutlet elements:

- `HeadLayout` / `ScriptLayout` - these allow a Razor layout (e.g. `MainLayout.razor`) to push site-wide CSS and scripts into the output HTML
- `HeadPage` / `ScriptPage` - these allow a Razor page (e.g. `Page.razor`) to push page-specific CSS and scripts into the output HTML
- `SitecoreEditingScripts` - the SDK itself writes to this outlet when the page is loaded inside the Pages editor. You do not push to it from your own components.

## Routes.razor

The default Blazor template generates a `Routes.razor` that uses the standard `<Router>` component. Replace it with the SDK's `<SitecoreRouter>` so that any URL not matched by an explicit `@page` directive falls through to Sitecore:

```html
<SitecoreRouter AppAssembly="typeof( Program ).Assembly" NotFoundPage="typeof( Pages.Status )">
    <Found Context="routeData">
        <RouteView RouteData="routeData" DefaultLayout="typeof( Layouts.MainLayout )" />
        <FocusOnNavigate RouteData="routeData" Selector="h1" />
    </Found>
</SitecoreRouter>
```

The `NotFoundPage` parameter is used when neither an explicit Blazor route nor a Sitecore item matches the requested URL. The Starter Kit points it at the `Status` page; see [Error handling](error-handling.md) for the details.

## MainLayout.razor

`MainLayout.razor` is the default layout for every Sitecore-routed page. Keep it small - most of the head and script content belongs in your `Page.razor` template so that it varies with the page type. A minimal layout looks like this:

```html
@inherits LayoutComponentBase

<SectionContent SectionName="HeadLayout">
    <!-- Add any other head content that should apply to all Sitecore pages -->
</SectionContent>

@Body

<!-- standard Blazor error bar -->
<div id="blazor-error-ui" data-nosnippet>
    An unhandled error has occurred.
    <a href="." class="reload">Reload</a>
    <span class="dismiss">🗙</span>
</div>

<SectionContent SectionName="ScriptLayout" />
```

The SDK injects its own editing chrome and editing scripts when needed. These are handled by the SDK-provided component `<SitecoreEditingScripts />` but you do not need to add this to the layout yourself.
